# Releasing — Operator Runbook

This page is for **maintainers**, not consumers. It captures the
mechanics of cutting a heliaCORE release and the lessons learned the
hard way during the first releases.

## The happy path

1. Merge feature/fix PRs to `main`. Subjects **must** be conventional
   commits (`feat:`, `fix:`, `feat!:`, …).
2. After a few PRs, release-please opens (or updates) a "Release PR"
   that bumps the version in `Ambiq.NS-CMSIS-NN.pdsc`,
   `.release-please-manifest.json`, and `CHANGELOG.md`.
3. **Review** the Release PR. The body shows every commit that will
   ship and the resulting version.
4. **Merge** the Release PR. release-please then:
     - creates `vX.Y.Z` tag and the GitHub Release,
     - the rest of `release.yml` builds and uploads all artifacts.
5. Watch the run. When green, the Release is consumer-ready.

## When things go sideways

### `publish-pack` fails with `xmlParseEntityRef: no name`

**Symptom**

```
/build/Ambiq.NS-CMSIS-NN.pdsc:23: parser error : xmlParseEntityRef: no name
gen_pack.sh> build aborted: Schema check ... against /tmp/PACK.xsd failed!
```

**Cause**

The GitHub Release body (auto-generated by release-please from your PR
titles) contains a character that's reserved in XML — typically `&`,
`<`, or `>`. `gen_pack.sh` was at some point configured with
`PACK_CHANGELOG_MODE=release`, which injects the release body verbatim
into the pdsc `<release>` element. Reserved chars escape XML parsing.

**Fix (forever)**

`gen_pack.sh` now uses `PACK_CHANGELOG_MODE="tag"` so the pdsc only
emits `<release version="…" date="…"/>` (no body). Full release notes
still live on the GitHub Release page.

**Fix (right now, for an already-broken release)**

1. Edit the GitHub Release body to replace the offending char
   (`&` → `and`, etc.):

   ```bash
   gh release view vX.Y.Z --json body -q .body > /tmp/body.md
   sed -i 's/&/and/g' /tmp/body.md
   gh release edit vX.Y.Z --notes-file /tmp/body.md
   ```

2. Re-run the failed jobs of the original run:

   ```bash
   gh run rerun <run-id> --failed
   ```

### `release-unit-tests` / `release-helia-core-tester` fail with `manifest unknown`

**Symptom**

Every matrix arch fails docker pull on
`ghcr.io/ambiqai/ns-cmsis-nn-ci:vX.Y.Z`.

**Cause**

`publish-ci-image` previously detected its trigger via
`github.event_name`. In a reusable workflow invoked with `uses:`,
`event_name` reflects the **caller's** event (`push`), not
`workflow_call`. The case statement fell through to default and only
pushed `:latest`.

**Fix (forever)**

`build_publish_docker.yml` now detects the workflow_call path by the
presence of `CALL_IMAGE_TAG` instead of by `event_name`.

**Fix (right now, for an already-broken release)**

Manually dispatch the docker build with the missing tag:

```bash
gh workflow run build_publish_docker.yml --ref main \
  -f image_tag=vX.Y.Z -f publish_latest=false
```

Once the tag is pushed:

```bash
gh run rerun <release-run-id> --failed
```

### A non-release-bumping commit landed and you want the next release sooner

Merge any commit with a `feat:` or `fix:` subject. release-please will
update its Release PR accordingly.

### You need to re-run a release **with code from `main`** (not the original SHA)

`gh run rerun` re-executes at the original commit. To pull in
post-release fixes, you need to either:

- Cut the next release (release-please will roll forward), or
- Add a `workflow_dispatch` path to the affected workflow and run it
  manually against `main` with the relevant inputs.

## Useful commands

```bash
# Watch the latest release.yml run
gh run watch "$(gh run list --workflow release.yml --limit 1 --json databaseId -q '.[0].databaseId')"

# See exactly what failed on a run
gh run view <run-id> --log-failed | grep -iE 'error|fail|##\[error\]' | head -40

# Re-run only failed jobs (preserves outputs of successful jobs)
gh run rerun <run-id> --failed

# Confirm a release has all expected assets
gh release view vX.Y.Z --json assets -q '.assets[].name' | sort
```
